# Правила оформления коммитов и PR(Pull Requests)

## Коммиты

Один коммит — одна небольшая завершённая задача.

**Формат сообщения** (Conventional Commits):
<тип>: <краткое описание>
[необязательно подробное объяснение]
**Основные типы (всегда на английском):**
- `feat` — новая возможность  
- `fix` — исправление ошибки  
- `refactor` — улучшение/переписывание кода без изменения поведения  
- `docs` — документация  
- `test` — тесты  
- `style` — форматирование, пробелы  
- `chore` — мелкие задачи (зависимости, настройки)

**Хорошие примеры коммитов:**
feat: добавлена стратегия случайного выбора ответа
fix: исправлена ошибка таймаута при отправке ответа
refactor: вынесен интерфейс стратегий в strategy.py
docs: обновлено описание архитектуры в ARCHITECTURE.md
test: добавлены тесты на fallback-логику
chore: обновлена версия pytest до 8.3

**Плохие примеры (не использовать):**
добавил стратегию
исправил баг
WIP
обновил файл
теперь работает

**Рекомендация:** перед созданием PR чисти историю — `git rebase -i HEAD~n`

## Pull Request

**Название PR** — тоже в стиле Conventional Commits:
feat: добавлена поддержка нескольких стратегий выбора ответа
fix: исправлена обработка ошибок сети в file.py
docs: добавлены правила оформления PR и коммитов
test: тесты на новую стратегию RandomChoice
**Описание PR** (обязательно заполнять):
Что сделано

Добавлена новая стратегия RandomStrategy
Добавлена настройка default_strategy в конфиге
Написаны тесты для всех стратегий
Исправлен баг с таймаутом запроса

**Зачем**
Раньше была только одна жёсткая стратегия.
Теперь можно легко подключать и тестировать разные алгоритмы выбора ответа.

**Как проверить**

`pip install -r requirements.txt`

`pytest`

`python main.py`
Пройти 10–15 вопросов → убедиться, что ответы выбираются по выбранной стратегии

- Хорошо: 100–400 строк
- Максимум: 600–800 строк
- Больше → лучше разбить на 2–3 PR
