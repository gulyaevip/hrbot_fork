# Архитектура проекта

## Общий подход
Проект реализуется с использованием простой модульной архитектуры.
Базовая логика приложения отделена от расширяемых компонентов.
Все дополнительные и экспериментальные возможности выносятся в отдельный каталог `modules`.

Архитектура ориентирована на:
- простоту понимания
- модульность
- расширяемость
- удобство сопровождения и code review

# Архитектура проекта

## Описание структуры каталогов

- **`src/`**  
  Основной каталог с исходным кодом проекта.  
  Содержит ядро приложения и расширяемые модули.

- **`src/modules/`**  
  Каталог расширяемых модулей.  
  Предназначен для размещения логики, которая может:  
  • изменяться  
  • дополняться  
  • экспериментально развиваться  
  без изменения базового кода приложения.
  - **`data/`** 
  Каталог для хранения данных приложения.
  Содержит файл базы данных SQLite.
- **`tests/`**  
  Каталог с автоматическими тестами.  
  Содержит модульные тесты для проверки корректности работы отдельных компонентов.

- **`docs/`**  
  Каталог с проектной документацией.  
  Используется для фиксации:  
  • архитектурных решений  
  • требований  
  • правил разработки


## Описание ключевых файлов

### `main.py`
**Точка входа в приложение**

- Запуск приложения
- Инициализация основных компонентов
- Подключение модулей

**Важно:** не содержит бизнес-логики.

### `bot.py`
**Основной управляющий модуль приложения**

- Управление процессом прохождения тестирования
- Взаимодействие с системой тестирования
- Использование модулей 

**Важно:** является центральной точкой координации всего приложения.

### `utils.py`
**Вспомогательный модуль общего назначения**

Может содержать:
- логирование
- обработку ошибок
- общие вспомогательные функции

**Важно:** не должен содержать бизнес-логику.

### `modules/try_module.py`
**Пример реализации расширяемого модуля**
- Демонстрация правил разработки модулей
- Шаблон / заготовка для создания новых модулей
- Базовая (примерная) стратегия выбора ответов


## Принцип расширяемости

Для добавления новой логики необходимо выполнить **три шага**:

1. Создать новый файл в каталоге `modules/`
2. Реализовать модуль по примеру определённому в `try_module.py`
3. Подключить модуль

Базовая логика приложения при этом **не изменяется**.

## Ограничения и правила

- `main.py` **не содержит** бизнес-логики
- `bot.py` — **единственная** точка координации и подключения модулей
- Расширяемая логика размещается **только** в каталоге `modules/`
- Каждый модуль должен быть **изолирован** и **независим**

## Тестируемость

Архитектура позволяет:

- Тестировать модули **изолированно**
- Использовать **заглушки** (mocks) для системы тестирования
- Проверять корректность работы **стратегий выбора ответов**
